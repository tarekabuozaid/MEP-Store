# الهندسة والخطة المقترحة — Google + واجهة موحدة + Gmail فقط

هذا المستند يلخص **ما اتُفِق تصميميًا على توثيقه** في المحادثة؛ التنفيذ الفعلي (Sheets/Web App) خارج إطار هذا التوثيق حتى يبدأ طرف تنفيذ فعلياً.

## القرارات المثبتة

| البند | الاختيار |
|-------|-----------|
| المنصّة الأساس المستهدف | Google Sheets + Google Apps Script |
| نموذج التشغيل | **خيار B**: ملف/جداول خلفية **مركزية** + **طبقة Apps Script/Web App** |
| الواجهة للموظَّفين | **واجهة موحَّدة واحدة** (رابط Web App واحد) |
| حسابات المستخدمين | **Gmail فقط**؛ لا اشتراك **Google Workspace** كمتطلّبات أولى |
| فصل المتاجر | جدول **`Users_Stores`** يَربط كل **بريد** بـ **`StoreCode`** و **`Role`** |
| حساب تنفيذ السكربت عند الكتابة | **`Execute as: Me`** بحساب **المالك / الأدمن** الذي نشر تطبيق الويب |
| نشر Web App (مبدئياً) | وصول بحيث تعمل **`Session.getActiveUser()`**؛ غالباً **`Anyone with Google account`**؛ تجنَّب الوصول المجهول إن كان تحديد المرسل مهماً لتدقيقكم |

## لماذا ليس مجرد “مشاركة شيت بلا سكربت”؟

**Google Sheets لا يمنح عمومًا قائمة تصريح “خلية/ورقة وحدة لمستخدم وحيد” بدون الهندسة المركّبة.**

من هنا تأتي الواجهة التي:

- لا تعتمد على إعطاء **تحرير** للمصنّف المركزي لجميع المتاجر؛
- تتحقَّق هوية المتصل بالبريد مقابل **`Users_Stores`** قبل السماح بقراءة البيانات الحساسة أو كتابة حركة عبر السكربت.

## مخطّط بيانات وظيفية (مسودّة)

- **`Master_Items`**, **`Locations`**, **`Stock_Movement`** حيث **كل صف حركة له `Location`** (إلزامي منطقيًا لفصل ستور)، **`Current_Stock`** أو View محسوب **حسب المواقع** حيث يلزم.
- **`Users_Stores`**: `Email`, `StoreCode`, `Role` (`Admin` | `Keeper` | دور قراءة مثلاً).
- المنطق البرمجي يعكس روح ماكرو **`SubmitEntry`**: Receipt / Issuance / Adjustment / Transfer؛ تحذير عند التدنّي تحت الحدّ الأدنى؛ منع الصرف أو التحويل فوق الرصيد المتاح لذلك المتجر؛ توليد **TxnID**؛ التحويل كزوج صفَّين بحسب ما اتُفِق عليه (مثلاً OUT عند المصدر ثم IN عند الوجهة ضمن **`Stock_Movement`**).

### مواءمة ما كان في Excel القديم

- صيَغ Excel القديمة قد تكون **مجمّعة بحسب الصنف دون الموقع** — إن كان ذلك لا يحقق فصل المتاجر المرغوبًا، يُعاد تنظيم الصيغ أو الاستعلام هنا ليفرِّق المواقع بحسب التصميم.- **VBA → Apps Script**.
- تجنَّب المراجع لمستودع عمود بالكامل في Google؛ استعمل **جدولاً مسمّى** وبنطاقاته الفعلية لـ **`SUMIFS`** أو استعلام سكربت محدَّد الصفوف.

## مخاطر ومحدودات Gmail (بدون Workspace)

- صلاحية كل مستخدم تعتمد على تحديث **`Users_Stores` يدوياً**؛ إذا كان الإيميل غير مسجّل تُعرض رسالة خطأ أوضح لعدم المصادقة.
- لا توجد مجموعات مؤسسية بنفس عمق Workspace؛ يمكن استخدام **مجموعات Google عامة** (`@googlegroups.com`) لتنظيم التواصل وليس بديل هوية آلية تقنية.
- **حصص Apps Script** للحسابات الاستهلاكية محدودة مقارنة بأعباء ثقيلة متزامنة.
- سياسات الأجهزة وMFA المركزية عبر المنظمة أقل مما لو كان لديكم اشتراك Business؛ اعتمدوا على سرّية الرابط، تدريب المستخدمين، وصلاحيات الملف المركز بحذر.

## خيارات احتياط (مرجع)

- **أ**: ملف Google لكل متجر + **`IMPORTRANGE`** للوحة تجميع؛ أبسط لكن يفرض سياسات مزامنة بين عدة مصنفات؛
- **ج**: وقت لاحق **BigQuery/Cloud SQL** للمعاملات عند احتياج أكبر؛ مازال خارج نطاق “Sheets وحده” الأولي.

## مراحل تسليم تشغيلية

1. تجمّيد نموذج الحقول + **`Users_Stores`**؛
2. **POC Gmail**: حسابان مختلفان في الجدول، Web App واحد؛ رفض حساب ثالث غير مسجل؛ قياس زمن الإرسال بما يُشبه الاستخدام الحقيقي؛
3. التوسعة لجميع المتاجر؛ إدارة الرابط ومشاركة الملف المركز بحذر؛
4. **هجرة بيانات** جزئية من Excel حسب الأولويات؛
5. تدقيق (Audit حسب المتاح ضمن المنصَّة وجداولكم).

## مهام متابعة (مرجع)

قائمة المهام المعنونة في مخطَّط Cursor (YAML) تمثل نفس نقاط العمل المتبقية: مخطّط الجداول، أمن النشر، POC السكربت، وضوابط توزيع الوصول تحت Gmail.
