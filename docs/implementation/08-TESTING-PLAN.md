## 3. اختبارات التكامل

### TC-INT-01: تدفق استلام كامل

```
السيناريو:
  Keeper يسجل استلام 10 قطعة ABC-001 في MZ

الخطوات:
  1. الـ client يُرسل payload صحيح
  2. AuthService يُقرّ الصلاحية
  3. validateHeader تنجح
  4. validateLines تنجح
  5. لا فحص رصيد (Receipt)
  6. generateTxnId يُنتج 'REC-2026-NNNN'
  7. writeRows يكتب صفاً واحداً
  8. AuditService يُسجّل

التحقق:
  ✅ Stock_Movement يحتوي الصف الجديد بكل الحقول
  ✅ Audit_Log يحتوي سجلاً بـ UserEmail صحيح
  ✅ الاستجابة: {success:true, txnId: 'REC-2026-NNNN', rowsWritten: 1}
  ✅ getBalance(ABC-001, MZ) بعد العملية = الرصيد السابق + 10
```

### TC-INT-02: تدفق صرف مرفوض

```
السيناريو:
  Keeper يحاول صرف 100 قطعة ABC-001 (الرصيد = 47)

التحقق:
  ✅ الاستجابة: {success:false, errors:[{row:1, message:'رصيد غير كافٍ...'}]}
  ✅ Stock_Movement لم يُكتب فيه أي صف
  ✅ Counters لم يتغير
```

### TC-INT-03: تدفق تحويل كامل

```
السيناريو:
  Admin ينقل 5 قطعة ABC-001 من MZ إلى L
  الرصيد في MZ = 47، في L = 23

التحقق:
  ✅ صفّان في Stock_Movement:
     - TRF-2026-NNNN-OUT | Location=MZ | Qty=5
     - TRF-2026-NNNN-IN  | Location=L  | Qty=5
  ✅ getBalance(ABC-001, MZ) = 47 - 5 = 42
  ✅ getBalance(ABC-001, L)  = 23 + 5 = 28
```

### TC-INT-04: تزامن معاملتين

```
السيناريو:
  Keeper1 وKeeper2 يرسلان معاملتين في نفس اللحظة

التحقق:
  ✅ كلا المعاملتين تُحفظ بنجاح
  ✅ TxnIDs مختلفان (LockService فعّال)
  ✅ لا تعارض في Stock_Movement
```

## 4. اختبارات القبول (UAT)

### UA-01: أمين مخزن يُسجّل استلاماً

```
المنفذ: أمين مخزن MZ
الخطوات:
  1. يفتح الرابط من موبايل
  2. يختار "استلام"
  3. يُدخل تاريخ اليوم، LPO
  4. يكتب أول 3 حروف من اسم الصنف → يختار من الاقتراحات
  5. يُدخل الكمية
  6. ينقر "معاينة" ثم "تأكيد"

المتوقع:
  ✅ يُحفظ بنجاح
  ✅ رسالة نجاح تظهر
  ✅ النموذج يُصفَّر
  ✅ يستطيع التحقق من الرصيد في شاشة "مخزوني"
```

### UA-02: أمين مخزن يحاول الوصول لمخزن آخر

```
المنفذ: Keeper MZ
الخطوات:
  1. يُحاول إرسال معاملة بـ sourceLocation='L'

المتوقع:
  ✅ الخطأ 'ACCESS_DENIED' يُرجَع
  ✅ لا شيء يُكتب
```

### UA-03: أدمن يرى Dashboard

```
المنفذ: Admin
المتوقع:
  ✅ يرى حركات اليوم لكل المخازن
  ✅ يرى الأصناف تحت الحد الأدنى
  ✅ يرى آخر 10 معاملات
```

### UA-04: أدمن يُضيف مستخدماً جديداً

```
المنفذ: Admin
الخطوات:
  1. إدارة → + إضافة
  2. يُدخل بريد جديد لم يُستخدَم من قبل
  3. يختار المخزن والدور
  4. يحفظ

المتوقع:
  ✅ يظهر في القائمة
  ✅ البريد الجديد يستطيع الدخول والعمل فوراً
```

### UA-05: مستخدم غير مسجل

```

المتوقع:
  ✅ صفحة رفض واضحة
  ✅ لا وصول لأي بيانات
```

## 6. سجل UAT الأولي — بيئة حقيقية (مايو 2026)

اختبارات يدوية عبر المتصفّح بعد نشر مشروع Apps Script «Aldhafra IMS» وربطه بمستودع MEP-Store (`clasp push`).  
للاستدلال وليس بديلاً عن إكمال باقي حالات هذا الملف.

| المرجع | ماذا تُحقّق | الحالة |
|--------|-------------|--------|
| فتح الويب أب `/exec` + الدخول بحساب مُصرَّح | تظهر الواجهة؛ Dashboard يحمّل | ✅ ناجح |
| Admin — شريط التنقل والتبويبات | Users / Items / Locations تظهر | ✅ ناجح |
| Items — إضافة صنف | مثلاً كود TEST-001، وحفظ، يظهر في الجدول | ✅ ناجح |
| New Transaction — استلام MZ | صنف TEST-001، كمية 100؛ ظهور TxnID (مثل REC-2026-0001) | ✅ ناجح |
| Stock — بعد الاستلام | رؤية الرصيد/الحركة المتوقعة لـ MZ وTEST-001 | ⏳ تحت المراجعة / إتمام المتبقي (انظر خطوات التحقّق أسفله) |

**إتمام صفّ «Stock» (مرجع واجهات: `06-UI-SCREENS.md` — §4 مخزوني الأمين؛ §8 المخزون الكامل للأدمن؛ و`09-DEPLOYMENT-GUIDE.md` المرحلة 4.3):**

1. بتسجيل دخول **حساب مُصرّح لمخزن MZ** (Keeper أو في حدود رؤية المخزون): افتح **«المخزون»** من شريط التنقل؛ تأكّد ظهور `TEST-001` وبأن الرصيد يعكس الاستلام (مثل +100 بعد معاملة الاستلام أعلاه).
2. افتح **الحركات / التاريخ** لنفس المخزن والصنف؛ تأكّد أن آخر القيود تُطابق `TxnID` الظاهر بعد حفظ الاستلام.
3. (اختياري لكن موصى به) في Google Sheets: راجع ورقة **`Stock_Movement`** بحثاً عن صف يطابق نفس **`TxnID`** و`ItemCode = TEST-001` و`Location = MZ` و`TxnType = Receipt` (قيمة `CONFIG.TXN_TYPES.RECEIPT` في `src/Config.gs`).

عند الإغلاق: حدّث عمود الحالة في الجدول أعلاه إلى **✅ ناجح** واذكر التاريخ والنسخة المنشورة (`/exec`) في سطر ملاحظة داخلية إن رغبت.
